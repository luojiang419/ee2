# 海军改造参考说明（E1–E10）

本文档用于后续继续修改海军时快速参考，只记录：
- 设计思路
- 关键修改位置
- 哪些位置改了不生效 / 会被别处覆盖
- 常用中英关键词对照

不记录具体数值表。

---

## 一、这轮海军改造的设计思路

### 1. 通用三船型基线

E6–E10 的通用三船型按下面的身份来做：

- **Frigate**：轻型快船 / 风帆护卫舰
- **FightingSail**：中型主力 / 风帆驱逐舰
- **Galleon**：重装主力 / 风帆战列舰

目标原则：

1. **时代升级要平滑**：E6→E7→E8→E9→E10 不能突然断崖跳变。
2. **越贵越值**：贵船的基础中性性价比略高。
3. **后期成长加速**：E8–E10 的成长和升级成本都应明显高于前期，不能平均每代加一点。
4. **成本比大致稳定**：
   - `Frigate : FightingSail : Galleon ≈ 1 : 2 : 5`
5. **克制关系按“同经济成本”成立**：
   - `Galleon > FightingSail`
   - `FightingSail > Frigate`
   - `Frigate > Galleon`

也就是说，不是简单地“面板最大就全赢”，而是：
- Galleon 更适合压中型船
- FightingSail 更适合压轻型船
- Frigate 以机动/效率反制重船

### 2. 早期海军（E1–E5）

- 早期主要锚点是 **WarGalley**。
- 作用是把 E1、E3、E5 的海军做成一个平滑过渡的早期体系。
- 它不是风帆三主船型的一部分，而是前置海军阶段。

### 3. 区域 / 文明特色船的原则

这轮的最终规则已经统一为：

- **区域特色船：E6–E10 统一按“同类型通用船的 1.5 倍造价”处理**
- **文明特色船：也按所属通用类型做 1.5 倍锚定**
- **MaChuan（马船）虽然是驱逐线，但也按同类型通用船 1.5 倍思路处理**

注意：
- 旧文档里曾有“区域船 E6–E7 按 1.2×，E8–E10 按 1.5×”的规则；
- 现在已经按用户最新要求，统一改为 **E6–E10 都按 1.5×**；
- 后续不要再按旧 1.2× 思路继续改。

### 4. 特色船不是“全面上位替代”

特色船要有明显性格，但不能全维度碾压：

- **MiddleEastBoat**：肉盾 / 续航
- **WesternBoat2 / AfricanBoat / MesoAmerBoat**：劫掠 / 爆发
- **JapanBoat2**：灵活 / 均衡 / 远射
- **ZhengHe**：肉盾旗舰
- **JapanBoat1**：灵活长射程旗舰
- **Guichuan**：极重装旗舰
- **WesternBoat1**：爆发型旗舰
- **MaChuan**：轻快火力线

### 5. 科技升级成本思路

`NavalAcademy` 的升级科技成本采用：

- **TECHPTS（科技点）为主成本**
- 普通/特殊资源为辅成本
- 资源种类尽量不要让不同舰种完全雷同
- 后期升级要更贵，且收益也更大

### 6. 特殊资源合法性必须遵守

以 `资源可用纪元说明.md` 为准：

- **Tin / 锡**：E1–E6
- **Iron / 铁**：E4–E9
- **Saltpeter / 硝石**：E7–E12
- **Oil / 石油**：E10–E15
- **Uranium / 铀**：E13–E15

做 E1–E10 海军时，重点就是：
- E6 还能用 Tin / Iron
- E7–E9 用 Iron / Saltpeter
- E10 才能出现 Oil

## 三、哪些地方“改了可能不生效”或会被别处覆盖

这是后续最需要注意的部分。

## 1. `upgrade_unittypes.csv` 不是唯一来源，DDF 的升级引用会覆盖一部分实际表现

有些船的最终表现不只取决于 CSV 面板，还会被 DDF 里的升级块影响。

### 已确认的重要覆盖/跳档点

#### `fighting_sail.ddf`
- `FightingSailEpoch8Size`
- `FightingSailEpoch9Size`
- `FightingSailEpoch10Size`

#### `galleon.ddf`
- `GalleonEpoch8Size`
- `GalleonEpoch10Size`

#### `Tao_Ship_FarEast.ddf`
- `ZhengHeEpoch6Attack`
- `ZhengHeEpoch10Attack`
- `ZhengHeEpoch6Size`
- `ZhengHeEpoch10Size`
- `MaChuanEpoch4Size`
- `MaChuanEpoch6Attack`
- `MaChuanEpoch6Size`
- `MaChuanEpoch10Attack`
- `MaChuanEpoch10Size`

#### `Tao_Ship_Western.ddf`
- `WesternBoat210Size`
- `WesternBoat110Size`
- 以及若干 `UpgradeAbilities` / `UpgradeSize`

### 含义

如果你只改 CSV：
- HP / 伤害 / 射程 / 装填 / 造价 这些会改；
- 但尺寸、某些攻击能力、甚至视觉/手感，可能仍被 DDF 中的升级块影响。

所以：
- **改面板和造价，看 CSV**
- **改单位身份、移动、尺寸、某些攻击行为，看 DDF**

## 2. `POP CAPACITY` 改了通常没用，真实人口看 DDF 的 `popCount`

对战船来说：
- `upgrade_unittypes.csv` 里的 `POP CAPACITY` 基本不是实际生效来源
- 实际人口看 DDF 里的：
  - `properties { popCount = N }`

所以如果你要改人口：
- 改 `frigate.ddf` / `galleon.ddf` / `fighting_sail.ddf` / `Tao_Ship_*.ddf`
- **不要只改 CSV 的 POP CAPACITY**

## 3. 船的“类型/克制身份”不是看中文名字，而是看 DDF 的 `rps`

例如：
- 你把某船文案写成“战列舰”，不代表它真的按战列舰逻辑参与体系
- 真正决定它归类的是：
  - `rps = Galleon`
  - `rps = FightingSail`
  - `rps = Frigate`
  - `rps = WarGalley`

所以如果要改“它属于哪类船”：
- 改 DDF 的 `rps`
- 不要只改 CSV 名字 / 图标 / 文案

## 4. 科技图标不是看单位文件，而是看 `dbtechtreenode.csv`

如果用户说：
- “海军研究院升级图标不对”

优先看：
- `EE2X_db/TechTree/dbtechtreenode.csv`

不要去单位 DDF 里找图标按钮来源。

## 5. 升级科技真正引用哪个升级项目，要看 `dbtechtreenode.csv` 的 `UPGRADE` 列

比如：
- `upGuichuan_8` 指向的是 `guichuanUpgradeEpoch8`

如果这里引用名不一致，即使单位行改了也可能不生效或升级不到。

## 6. 资源合法性不能只看“像不像合理”，必须按纪元规则查

尤其容易出错的是：
- E6 误用 Saltpeter
- E7–E9 误保留 Tin
- E10 之前误用 Oil

所以海军改完后，最好重新检查：
- `upgrade_unittypes.csv`
- `dbtechtreenode.csv`

确保资源列没错位、没越时代。

## 六、后续继续改海军时的建议顺序

如果以后继续调海军，建议顺序是：

1. **先定设计目标**
   - 是改通用三船型？
   - 还是改特色船偏置？
   - 还是改升级成本？

2. **先改 `upgrade_unittypes.csv`**
   - 先把单位本体数值和造价定下来

3. **再改 `dbtechtreenode.csv`**
   - 让升级科技成本、图标、升级引用跟上

4. **最后检查 DDF**
   - 看 `rps`、`popCount`、`NavalMove`
   - 看 `UpgradeAbilities` / `UpgradeSize` 有没有覆盖或跳档

5. **最后再看文档是否与当前规则一致**
   - 尤其注意不要把旧的 1.2× 规则又写回来
