## 1.5 多机器自动同步机制

> **本仓库在多台机器上通过同一 GitHub 账号开发，每次操作必须保证数据同步。**

### 1.5.1 🔴 会话开始强制拉取（硬门禁）

> **拉取未成功 = 禁止任何开发操作。这是不可跳过的硬门禁。**

- **每次新会话读到本文档时，第一件事就是执行 `git pull origin main`**，在所有其他操作之前
- **必须确认拉取成功**（输出 `Already up to date` 或 `Fast-forward` 合并成功）后，才能继续任何开发工作
- **如果拉取失败**（网络错误、冲突、认证失败等），必须立即停止，先解决拉取问题，**严禁在未同步状态下修改任何文件**
- 这是"第 0 步"，优先级高于所有其他任务，无例外

### 1.5.2 提交后自动推送
- 已配置 post-commit hook（`.git/hooks/post-commit`），每次 `git commit` 后自动 `git push origin main`
- 推送失败时终端会提示，需手动执行 `git push origin main`

### 1.5.3 🔴 强制推送铁律（每次修改必须推送）
- **每次修改完成后，无论改动多少（哪怕只改一行），必须立即 `git add` + `git commit` + 推送到 GitHub**
- **严禁累积多次修改后再一次性提交推送**，累积提交会导致：多机器版本不同步、冲突风险增加、修改历史模糊、回滚粒度太粗
- 每次修改 = 一个独立 commit，commit message 应精确描述本次改动内容
- 此规则无例外，包括：数值微调、单行修正、临时修复、测试性修改

```
强制推送工作流:
  修改完成 → git add → git commit → 自动push → GitHub
  ↓
  下一个修改 → 重复上述流程
  ↓
  （禁止）修改A + 修改B + 修改C → 一次性commit
```

```
多机器工作流:
  机器A → 修改 → commit(自动push) → GitHub
                                          ↓
  机器B → 读本文档 → git pull(自动) → 同步最新代码
```

## 二、任务协作规则

### 2.1 工作模式
1. **先规划后执行**: 接到任务后先做整体规划，输出任务目标、拆分步骤、文件/模块清单、待办清单、验收标准
2. **小步推进**: 每次只完成一小部分，完成后告知当前进度，不一次性输出超长内容
3. **模块化开发**: 使用模块化结构，开发新功能模块后需全局检测是否整体正常运行
4. **全程中文**: 所有交流使用中文

### 2.2 进度快照制度
- 每完成一个功能修改，自动生成一份进度快照到 `进度快照\`
- 文件名格式: `{序号}-{简要描述}.md`，序号纯递增不回改
- 快照内容: 已完成内容 + 当前修改到的模块 + 待办清单（未完成）+ 下一步要做什么
- **新对话/恢复工作时**: 只需读取最新一份快照（按序号最大），不读取全部历史快照

### 2.3 中断恢复
- 如果新开对话、刷新页面、更换设备，自动读取 `进度快照\` 中序号最大的文件
- 直接从上一轮结束的位置继续，不再重复提问确认
- 同时参考 `单位修改日志\` 了解之前的修改历史

### 2.5 源码备份

#### 2.5.1 备份范围铁律

**两种备份各有用途，缺一不可：**

| 备份类型 | 时机 | 备份内容 | 用途 |
|:--------|:-----|:--------|:-----|
| **源文件备份** (pre) | 修改**前** | 本次即将修改的源文件 | 精确回退单个修改 |
| **游戏ZIP备份** (runtime) | 修改应用**后** | 完整 `EE2X_db.zip` | 快速恢复完整运行态 |

**源文件备份规则**：
- ✅ 正确: 修改 `dapao.ddf` + `upgrade_unittypes.csv` → 只打包这 2 个文件
- ❌ 错误: 无论改什么，都把整个 `EE2X_db\` 或整个项目目录打包进去
- ❌ 错误: 将 Flutter 构建产物、启动器 node_modules、游戏 NIF 模型等无关文件混入备份
- 理由: 精准打包便于回退时快速定位、体积小可推送 GitHub、索引描述也精确

**游戏ZIP备份规则**（详见 2.5.5）：
- 每次修改应用到 `Empire Earth II\zips_ee2x\EE2X_db.zip` 后，**立即备份整个ZIP**
- 这是运行态快照，游戏直接读取此ZIP，备份后可在任何时刻直接恢复游戏到该状态
- **严禁跳过此步骤**，这是防止黑盒修改崩溃无法恢复的最后防线

#### 2.5.2 备份格式
- **必须压缩为 `.zip` 包**，严禁直接复制裸文件夹到 `backup\`
- 命名格式: `EE2X_db-{NNN}-{中文简要描述}-{type}.zip`
  - NNN 为 3 位递增序号，自动分析 `backup\` 中已有备份的最大序号 +1
  - **中文简要描述**：用中文体现改什么单位、改什么属性，对中国用户直观可读
  - type: `pre` = 源文件备份 / `runtime` = 游戏ZIP运行态备份
  - 示例: `EE2X_db-022-火炮阵地E14火箭弹-pre.zip`、`EE2X_db-023-超级主力舰修改后完整ZIP-runtime.zip`
- 同一阶段内的多次小修改不重复备份
- ZIP 内部路径结构应与 `game-metadata\EE2X_db\` 保持一致，解压后直接覆盖即可

#### 2.5.3 备份索引（强制）

每次创建备份包后，**必须同步更新** `backup\INDEX.md`。每个条目必须包含以下 8 项信息：

```markdown
# 备份索引

> 最后更新: YYYY-MM-DD HH:MM

| 序号 | 备份包 | 类型 | 修改目标 | 包含文件 | 修改描述 | 运行状态 | 时间 |
|:-----|:------|:-----|:--------|:--------|:--------|:--------|:-----|
| 022 | EE2X_db-022-火炮阵地E14火箭弹-pre.zip | pre | dapao(火炮阵地) E14 | dapao.ddf, upgrade_unittypes.csv | 炮弹从DapaoShell改为MlrsRocket_PHL03火箭弹 | — | 2026-05-21 11:19 |
| 023 | EE2X_db-023-火炮阵地修改后完整ZIP-runtime.zip | runtime | 游戏运行态快照 | EE2X_db.zip 完整 | 022修改应用后的完整游戏ZIP | 待测试 | 2026-05-21 11:25 |
```

**字段说明**:
- **序号**: 递增编号，与 ZIP 文件名 NNN 一致
- **类型**: `pre` = 源文件修改前备份 / `runtime` = 游戏ZIP完整运行态备份
- **修改目标**: 改了哪个单位/建筑的什么时代什么属性
- **包含文件**: ZIP 内的文件列表（`game-metadata\EE2X_db\` 相对路径，runtime类型填写 `EE2X_db.zip 完整`）
- **修改描述**: 一句话说清改了什么、为什么改
- **运行状态**（🔴 必填）:
  - `待测试` — 刚备份完，尚未启动游戏验证
  - `测试通过` — 用户启动游戏验证无崩溃、功能正常
  - `崩溃` — 游戏启动/运行崩溃，需回退
  - `异常({描述})` — 游戏能运行但有异常（如数值不对、单位消失等）
- **时间**: 精确到分钟

**运行状态更新规则**:
- 创建备份时默认填写 `待测试`
- 用户反馈测试结果后，**必须立即更新**对应条目的运行状态
- 状态为 `崩溃` 或 `异常` 的备份不得作为后续修改的基准
- 只有 `测试通过` 的 runtime 备份才是可信的回退点

INDEX.md 纳入 Git 版本管理。

#### 2.5.4 备份回退流程

**源文件备份回退**:
1. 打开 `backup\INDEX.md` 查找需要回退到的备份包
2. 将对应 `.zip` 解压到 `game-metadata\EE2X_db\` 直接覆盖
3. 重新打包游戏 ZIP（`Empire Earth II\zips_ee2x\EE2X_db.zip`）
4. 启动游戏验证回退效果

**游戏ZIP快速回退**（推荐优先使用）:
1. 打开 `backup\INDEX.md`，找到最近一个 `测试通过` 的 `runtime` 备份
2. 将备份 ZIP 中的 `EE2X_db.zip` 直接覆盖到 `Empire Earth II\zips_ee2x\`
3. 启动游戏即可恢复到最后验证通过的状态
4. **无需重新打包**，runtime备份是完整游戏ZIP的直接拷贝

#### 2.5.5 🔴 游戏ZIP运行态备份（每次修改后强制执行）

> 这是防止黑盒修改崩溃无法恢复的**最后防线**，严禁跳过。

**时机**: 每次将修改应用到 `Empire Earth II\zips_ee2x\EE2X_db.zip` 之后，**立即**备份。

**步骤**:
1. 确认游戏ZIP已更新（修改已打包进去）
2. 将 `Empire Earth II\zips_ee2x\EE2X_db.zip` 复制到 `backup\`
3. 命名格式: `EE2X_db-{NNN}-{简要描述}-runtime.zip`
   - NNN 为递增序号（与源文件 pre 备份共用序号序列）
   - 示例: `EE2X_db-023-dapao-rocket-runtime.zip`
4. 更新 `backup\INDEX.md`，类型填写 `runtime`，运行状态填写 `待测试`
5. 此备份包纳入 Git 版本管理（`*.zip` 已在 .gitignore 白名单中）

**与 pre 备份的关系**:
```
修改前: 创建 pre 备份（源文件级，便于单个修改回退）
  ↓
修改源文件 + 打包游戏ZIP
  ↓
修改后: 创建 runtime 备份（完整游戏ZIP，便于整包快速恢复）
  ↓
用户测试反馈 → 更新 INDEX.md 运行状态
```

**回退时优先使用 runtime 备份**: 直接覆盖游戏ZIP即可恢复，无需重新打包。

## 四、关键数据链

### 4.1 文明属性生效链路（5步）
```
Civilizations/*.ddf (CivAttributes列表)
  → dbtechtreenode.csv (NAME=CivAtt_*, PRODUCE=Traits_*)
  → civ_reg_bonus_upgrades.ddf (UpgradeRefSet)
  → upgrade_factorset.csv (因子/方法/目标/数值)
  → dbunittypeattributes.ddf (目标单位类型族)
```

### 4.2 单位升级生效链路
```
dbtechtreenode.csv (UPGRADE列指向升级名)
  → upgrade_unittypes.csv (升级名对应的具体属性值)
  → Units/*.ddf (UpgradeAbilities/UpgradeSize可覆盖CSV值)
```

---

## 五、重要注意事项

- **CSV ≠ 唯一数据源**: DDF 中的 `UpgradeAbilities` 和 `UpgradeSize` 块会覆盖 CSV 同名属性
- **船只人口在DDF**: CSV 中的人口列对船只无效，实际消耗在 DDF 的 `popCount` 字段
- **RPS类型在DDF**: 单位的石头剪刀布类型由 DDF 的 `rps` 字段决定
- **升级引用一致性**: `dbtechtreenode.csv` 的 UPGRADE 值必须能在 `upgrade_unittypes.csv` 中找到，否则静默失效
- **资源时代限制**: Tin(E1-E6), Iron(E4-E9), Saltpeter(E7-E12), Oil(E10-E15), Uranium(E13-E15)
- **工作数据库不完整**: 工作 DB 无 Simulation 目录，RPS 克制修改需从完整解包 ZIP 提取
- **两个升级表**: `upgrade_unittypes.csv` (主表) 和 `upgrade_unittypes_EE2X.csv` (资料片专表) 并存
- **文明文件名**: 是 `MesoAmericanCivs.ddf` 不是 `MesoAmerCivs.ddf`
- **EXE有硬编码**: 某些常量和文明属性白名单在 `EE2X.exe` 中，仅改数据文件可能不完全生效
- **🔴 游戏读ZIP不读工作目录**: 修改 `game-metadata\EE2X_db\` 后必须更新 `Empire Earth II\zips_ee2x\EE2X_db.zip`，否则修改无效。详见 `地2方案文档\数据库修改生效链路与ZIP同步规则.md`
- **文本同步**: 修改选项/枚举数量时需同步更新 `DbFrontEndDefs.h` 和所有 `dbtext_*.utf8` 文件
- **🔴 单位名称铁律**: 无论任何场景（方案文档、进度快照、修改日志、对话交流），必须使用游戏文件内的**原始中/英文名称**，严禁自行翻译或创造译名。例如游戏中 `ArmoredCar` 就叫 ArmoredCar，不能写成"装甲车"；`LightTank` 不能写成"轻型坦克"。擅自翻译会导致查找游戏文件时找不到、其他人无法定位、修改时定位错误。唯一例外：单位修改日志的目录和文件名可使用中文简称以便查阅，但正文内容必须使用原始名称

### 5.1 DDF修改铁律（关键！违反必导致游戏崩溃）

> 详见 `地2方案文档/DDF与CSV安全修改规则.md`

- **DDF换行符必须是LF**: 绝不使用 `Set-Content`、`Out-File`、`WriteAllLines` 写入DDF文件，它们会自动转换CRLF导致游戏 ACCESS VIOLATION 崩溃
- **禁止正则跨行替换**: DDF中重复模式极多（如 `NavalMove {speed = 1.7` 出现44次），`(?s)` 跨行匹配会跨UnitType块误伤无关单位
- **只使用精确行号定位**: 先 `diff` 获取目标行号，用 PowerShell `ReadAllText` + `` "`n" `` split + 精确行替换 + `WriteAllText($false)` 保存
- **改后必须diff验证**: 每次修改后与备份对比，确认只有目标行被改动

### 5.3 修改安全流程（8步）

0. **🔴 先查总索引**: 按5.2的AI指令或人类指令执行，获取目标单位的全部源文件路径+精确行号
1. **备份源文件 (pre)**: 将本次即将修改的文件打包备份到 `backup\`（按 2.5.1 源文件备份规则）
2. **修改**: CSV用 Edit 工具、DDF用精确行号 + LF保持法（行号直接来自索引步骤0，无需再搜索）
3. **验证diff**: 修改后 diff 对比备份确认无误伤
4. **🔴 打包ZIP**: 将修改后的文件打包到游戏ZIP（`Empire Earth II\zips_ee2x\EE2X_db.zip`），否则游戏不生效。详见 `地2方案文档\数据库修改生效链路与ZIP同步规则.md`
5. **🔴 备份游戏ZIP (runtime)**: 将完整的 `EE2X_db.zip` 复制到 `backup\`，更新 INDEX.md，运行状态填 `待测试`（按 2.5.5 规则）
6. **提交推送**: `git add` + `git commit`，系统自动推送到 GitHub
7. **🔴 等待用户测试反馈（强制）**: 
   - **必须明确要求用户启动游戏测试**，告知用户测试哪些具体单位/功能
   - **严禁在收到用户反馈前开始下一轮修改**，这是防止黑盒修改累积崩溃的铁律
   - 用户反馈结果后，**立即更新** `backup\INDEX.md` 中对应 runtime 备份的运行状态
   - 只有用户确认 `测试通过` 后，才能继续下一轮修改
   - 如果用户反馈 `崩溃` 或 `异常`，优先使用 runtime 备份直接恢复游戏ZIP，再排查问题

```
修改安全流程（8步）:
  步骤0: 查总索引 → 步骤1: 备份源文件 → 步骤2: 修改 → 步骤3: diff验证
  → 步骤4: 打包游戏ZIP → 步骤5: 备份游戏ZIP → 步骤6: git提交推送
  → 步骤7: 🔴 等用户测试反馈（收到反馈前严禁继续）
```

> **为什么步骤7是铁律？** 之前黑盒开发连续修改多次后才测试，数据混乱导致游戏崩溃，且无 runtime 备份无法恢复。此后每次修改 = 一次完整的"修改→备份→测试→确认"闭环。

### 5.4 自动推送机制

2. **DDF完整参数表** — 中文标注+英文原名，覆盖DDF中出现的所有参数
3. **数据来源位置表** — 精确到每个文件的**行号**：
   - DDF属性（文件+行号）
   - CSV升级行（文件+行号）
   - 科技树节点（文件+行号）
   - 时代升级集（文件+行号，可能有多个引用）
4. **CSV升级数据** — E{时代}行的完整数值和资源造价
5. **科技树造价** — dbtechtreenode.csv中的建造资源+时间
6. **通用单位汇总表** — 该文明可用的All-civ通用单位（含全资源造价列）

> 详细格式见 `地2方案文档\单位属性文档模板.md`，该模板包含每种单位类型的完整示例。

### 7.6 使用方式

**查阅属性**:
- 想知道某单位的完整属性 → 打开 `单位属性文档\{文明}\{文明}_{时代}_{兵种}_单位属性.md`
- 想知道某属性的来源文件 → 查看"数据来源位置"表，文件名+行号一目了然

**快速定位修改**:
- 根据"数据来源位置"表直接跳转到目标文件的目标行
- 无需再次搜索、扫描、grep

**修改后更新**:
- 如果修改了数值，同步更新文档中对应的参数值
- 如果新增了单位，按模板格式追加新条目

### 7.7 数据提取工作流

```
1. CSV搜索  → grep "{UnitType}UpgradeEpoch{时代}" upgrade_unittypes.csv → 记录行号+数值
2. DDF定位  → grep "UnitType {UnitType}" Units/*.ddf → 读取properties/abilities块 → 记录行号+参数
3. 科技树   → grep "^{UnitType}," dbtechtreenode.csv → 记录行号+造价
4. 升级集   → grep "{UpgradeName}" epoch{时代}_upgrades.ddf → 记录所有引用行号
5. 写入文档 → 使用模板格式逐单位填充
```

### 9.1 工具使用注意事项

- **中文目录必须用 Bash `ls`**: `进度快照\`、`单位修改日志\`、`单位属性文档\` 等中文命名目录，Glob 和 PowerShell `Get-ChildItem` 均可能因编码问题返回空结果。必须使用 Bash 工具执行 `ls "中文目录名/"` 列出文件
- **Glob 可用于英文/数字路径**: `EE2X_db\`、`game-metadata\`、`backup\` 等纯 ASCII 路径可正常使用 Glob
- **Grep 中文内容搜索**: 搜索中文文件名或中文文件内容时，使用 PowerShell `Select-String` 或 Grep 工具均可，不受影响
